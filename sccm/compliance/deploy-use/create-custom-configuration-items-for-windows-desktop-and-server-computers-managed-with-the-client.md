# <a name="create-custom-configuration-items-for-windows-desktop-and-server-computers-managed-with-the-configuration-manager-client"></a>구성 관리자 클라이언트에서 관리되는 Windows 데스크톱 및 서버 컴퓨터에 대한 사용자 지정 구성 항목 만들기

*적용 대상: Configuration Manager(현재 분기)*


Configuration Manager **사용자 지정 Windows 데스크톱 및 서버** 구성 항목을 사용하여 구성 관리자 클라이언트에서 관리되는 Windows 컴퓨터 및 서버에 대한 설정을 관리합니다.  



## <a name="start-the-wizard"></a>마법사 시작

1. Configuration Manager 콘솔에서 **자산 및 준수** 작업 영역으로 이동하여 **준수 설정**을 확장하고 **구성 항목** 노드를 선택합니다.  

2. 리본의 **홈** 탭에 있는 **만들기** 그룹에서 **구성 항목 만들기**를 선택합니다.  

3. **구성 항목 만들기 마법사** 의 **일반**페이지에서 구성 항목에 대한 이름 및 선택적 설명을 지정합니다.  

4. **만들려는 구성 항목의 유형 지정**에서 **Windows 데스크톱 및 서버(사용자 지정)** 를 선택합니다.  

    > [!TIP]  
    > 애플리케이션이 있는지 여부를 확인하는 검색 방법 설정을 제공하려는 경우 **이 구성 파일에 애플리케이션 설정 포함**을 선택합니다.  

5. Configuration Manager 콘솔에서 구성 항목을 검색하고 필터링하려면 **범주**를 선택하여 범주를 만들고 할당합니다.  



## <a name="detection-methods"></a>검색 방법  

이 절차를 사용하여 구성 항목에 대한 검색 방법 정보를 제공합니다.  

> [!NOTE]  
> 이 정보는 마법사의 **일반** 페이지에서 **이 구성 항목에 애플리케이션 설정 포함**을 선택한 경우에만 적용됩니다.  

Configuration Manager의 검색 방법에는 컴퓨터에 애플리케이션이 설치되어 있는지를 검색하는 규칙이 포함됩니다. 이 검색은 클라이언트에서 구성 항목에 대 한 호환성을 평가 하기 전에 발생 합니다. 애플리케이션에 대 한 Windows Installer 파일의 존재를 감지할 수 애플리케이션이 설치 되어 여부를 검색 하려면 사용자 지정 스크립트를 사용 하거나 선택 **항상 애플리케이션이 설치 되어 있다고 가정** 애플리케이션 설치 여부에 관계 없이 규정 준수에 대 한 구성 항목을 평가할 수 있습니다.  


### <a name="to-detect-an-application-installation-by-using-the-windows-installer-file"></a>Windows Installer 파일을 사용하여 애플리케이션 설치를 검색하려면  

1. **구성 항목 만들기 마법사**의 **검색 방법** 페이지에서 **Windows Installer 검색 사용** 옵션을 선택합니다.  

2. **열기**를 선택하고 검색하려는 Windows Installer(.msi) 파일을 찾은 후 **열기**를 선택합니다.  

3. **버전** 필드에 Windows Installer 파일의 버전 번호가 자동으로 채워집니다. 표시 된 값이 잘못 된 경우 여기에 새 버전 번호를 입력 합니다.  

4. 컴퓨터에서 각 사용자 프로필을 검색하려는 경우 **이 애플리케이션은 하나 이상의 사용자에게 설치됨**을 선택합니다.  


### <a name="to-detect-a-specific-application-and-deployment-type"></a>특정 애플리케이션 및 배포 유형을 검색하려면  

1. **구성 항목 만들기 마법사** 의 **검색 방법** 페이지에서 **특정 애플리케이션 및 배포 유형 검색**을 선택합니다. **선택**을 선택합니다.   

2. **애플리케이션 지정** 대화 상자에서 검색하려는 애플리케이션 및 연결된 배포 유형을 선택합니다.  


### <a name="to-detect-an-application-installation-by-using-a-custom-script"></a>사용자 지정 스크립트를 사용 하 여 애플리케이션 설치를 검색 하려면  

1. **구성 항목 만들기 마법사**의 **검색 방법** 페이지에서 **이 애플리케이션을 검색하기 위한 사용자 지정 스크립트 사용** 옵션을 선택합니다.  

2. 목록에서 스크립트 언어를 선택 합니다. 다음 형식 중에서 선택합니다.  

    - **VBScript**  

    - **JScript**  

    - **PowerShell**  

        > [!Note]  
        > 버전 1810부터, Windows PowerShell 스크립트를 검색 방법으로 실행하면 구성 관리자 클라이언트는 `-NoProfile` 매개 변수를 사용하여 PowerShell을 호출합니다. 이 옵션은 프로필 없이 PowerShell을 시작합니다. PowerShell 프로필은 PowerShell이 시작될 때 실행되는 스크립트입니다. <!--3607762-->  

3. **열기**를 선택하고 사용하려는 스크립트를 찾은 다음 **열기**를 선택합니다.  



##  <a name="specify-supported-platforms"></a>지원되는 플랫폼 지정  

**구성 항목 만들기 마법사** 의 **지원되는 플랫폼** 페이지에서 구성 항목의 준수 여부를 평가하려는 Windows 버전을 목록에서 선택하거나 **모두 선택**을 선택합니다. 

**Windows 버전을 수동으로 지정할**수도 있습니다. **추가** 를 선택 하 고 Windows 빌드 번호의 각 부분을 지정 합니다. 



##  <a name="configure-settings"></a>설정 구성  

이 절차를 사용하여 구성 항목의 설정을 구성합니다.  

설정은은 비즈니스 또는 클라이언트 디바이스에서 호환성을 평가 하는데 사용 되는 기술적 조건을 나타냅니다. 참조 컴퓨터에서 기존 설정을 찾거나 새 설정을 구성할 수 있습니다.  

1. **구성 항목 만들기 마법사**의 **설정** 페이지에서 **새로 만들기**를 선택합니다.  

2. 에 **일반** 탭은 **설정 만들기** 대화 상자에서 다음 정보를 제공 합니다.  

    - **이름**: 설정에 대해 고유한 이름을 입력합니다. 최대 256자까지 사용할 수 있습니다.  

    - **설명**: 설정에 대한 설명을 입력합니다. 최대 256자까지 사용할 수 있습니다.  

    - **설정 유형**: 목록에 나온 다음 설정 유형 중에서 이 설정에 사용할 설정을 선택하여 구성합니다.  
        - [Active Directory 쿼리](#bkmk_adquery)
        - [어셈블리](#bkmk_assembly)
        - [파일 시스템](#bkmk_file)
        - [IIS 메타베이스](#bkmk_iis)
        - [레지스트리 키](#bkmk_regkey)
        - [레지스트리 값](#bkmk_regval)
        - [스크립트](#bkmk_script)
        - [SQL 쿼리](#bkmk_sql)
        - [WQL 쿼리](#bkmk_wql)
        - [XPath 쿼리](#bkmk_xpath)

    - **데이터 형식**: 목록에서 조건이 설정을 평가하는 데 사용되기 전에 조건이 데이터를 반환하는 형식을 선택합니다. 모든 설정 유형에 대해 **데이터 형식** 목록이 표시되는 것은 아닙니다.  

        > [!Tip]  
        > **부동 소수점** 데이터 형식은 소수점 이하 3자리만 지원합니다.  

3. 구성에서이 설정에 대 한 추가 세부 정보는 **유형을 설정** 목록입니다. 구성할 수 있는 항목은 선택한 설정 유형에 따라 달라집니다.  

4. **확인**을 선택하여 설정을 저장하고 **설정 만들기** 대화 상자를 닫습니다.  


### <a name="bkmk_adquery"></a> Active Directory 쿼리

- **LDAP 접두사**: 클라이언트 컴퓨터에서 호환성을 평가하도록 Active Directory Domain Services 쿼리에 올바른 접두사를 지정합니다. 글로벌 카탈로그 검색을 수행 하려면 `LDAP://` 또는 `GC://`를 사용 합니다.  

- **DN(고유 이름)** : 클라이언트 컴퓨터에서 호환성을 평가한 Active Directory Domain Services 개체의 고유 이름을 지정합니다.  

- **검색 필터**: 클라이언트 컴퓨터에서 호환성을 평가하기 위해 Active Directory Domain Services 쿼리의 결과를 구체화하기 위한 선택적인 LDAP 필터를 지정합니다. 쿼리의 모든 결과를 반환하려면 `(objectclass=*)`을 입력합니다.  

- **검색 범위**: Active Directory Domain Services에서 검색 범위를 지정합니다.  

    - **기준**: 지정된 개체만 쿼리합니다.  

    - **한 수준**: 이 옵션은 이 버전의 Configuration Manager에서 사용되지 않습니다.  

    - **하위 트리**: 디렉터리에서 지정한 개체와 전체 하위 트리를 쿼리합니다.  

- **속성**: 클라이언트 컴퓨터에서 호환성을 평가하는데 사용하는 Active Directory Domain Services 개체의 속성을 지정합니다.  

    예를 들어 사용자가 암호를 잘못 입력한 횟수를 저장하는 Active Directory 속성을 쿼리하려는 경우 이 필드에 `badPwdCount`를 입력합니다.  

- **쿼리**: **LDAP 접두사**, **DN(고유 이름)** , **검색 필터**(지정된 경우) 및 **속성**의 항목에서 구성된 쿼리를 표시합니다.  


### <a name="bkmk_assembly"></a> 어셈블리

어셈블리는 애플리케이션 간에 공유할 수 있는 코드 조각입니다. 어셈블리는 파일 이름 확장명 .dll 또는 .exe를 가질 수 있습니다. 전역 어셈블리 캐시는 클라이언트 컴퓨터에서 `%SystemRoot%\Assembly` 폴더입니다. 이 캐시는 Windows에서 모든 공유 어셈블리를 저장 하는 위치입니다.  

- **어셈블리 이름:** 검색할 어셈블리 개체의 이름을 지정합니다. 이름은 동일한 유형의 다른 어셈블리 개체와 같을 수 없습니다. 먼저 전역 어셈블리 캐시에 등록 합니다. 어셈블리 이름의 길이는 최대 256자일 수 있습니다.  


### <a name="bkmk_file"></a> 파일 시스템

- **형식**: 목록에서 **파일**을 검색할지, **폴더**를 검색할지를 선택합니다.  

- **경로**: 클라이언트 컴퓨터에서 지정된 파일 또는 폴더의 경로를 지정합니다. 경로에 시스템 환경 변수와 `%USERPROFILE%` 환경 변수를 지정할 수 있습니다.  

    > [!NOTE]  
    > **경로** 또는 **파일 또는 폴더 이름** 상자에 `%USERPROFILE%` 환경 변수를 사용 하는 경우 Configuration Manager 클라이언트는 클라이언트 컴퓨터의 모든 사용자 프로필을 검색 합니다. 이 동작으로 인해 파일이 나 폴더의 여러 인스턴스를 찾을 수 있습니다.  
    >   
    > 규정 준수 설정에서 지정된 경로에 액세스할 수 없는 경우 검색 오류가 생성됩니다. 또한 검색 하는 파일이 현재 사용 중인 경우에 검색 오류가 생성 됩니다.  

    > [!Tip]  
    > **찾아보기** 를 선택 하 여 참조 컴퓨터의 값에서 설정을 구성 합니다.   

- **파일 또는 폴더 이름**: 검색할 파일 또는 폴더 개체의 이름을 지정합니다. 파일 또는 폴더 이름에 시스템 환경 변수와 `%USERPROFILE%` 환경 변수를 지정할 수 있습니다. 파일 이름에 와일드카드 `*` 및 `?`를 사용할 수도 있습니다.  

    > [!NOTE]  
    > 파일 또는 폴더 이름을 지정할 때 와일드카드를 함께 조합해서 사용할 경우 많은 수의 결과가 반환될 수 있습니다. 또한 이로 인해 클라이언트 컴퓨터의 리소스 사용량이 증가하고 Configuration Manager에 결과를 보고할 때 네트워크 트래픽이 늘어날 수 있습니다.  

- **하위 폴더 포함**: 지정 된 경로 아래의 하위 폴더도 모두 검색 합니다.  

- **이 파일 또는 폴더가 64 비트 응용 프로그램과 연결 되어**있습니다. 사용 하도록 설정 하면 64 비트 컴퓨터에서 `%ProgramFiles%`와 같은 64 비트 파일 위치만 검색 합니다. 이 옵션을 사용 하지 않는 경우 `%ProgramFiles(x86)%`와 같은 32 비트 위치 및 64 비트 위치를 검색 합니다.  

    > [!NOTE]  
    > 동일한 64비트 컴퓨터의 64비트 및 32비트 시스템 파일 위치에 같은 파일 또는 폴더가 모두 있는 경우 글로벌 조건을 통해 여러 파일이 검색됩니다.  

    **파일 시스템** 설정 유형을 사용할 경우 **경로** 상자에서 네트워크 공유의 UNC 경로를 지정할 수 없습니다.  


### <a name="bkmk_iis"></a> IIS 메타베이스

- **메타베이스 경로**: IIS(인터넷 정보 서비스) 메타베이스의 유효한 경로를 지정합니다. 예: `/LM/W3SVC/`  

- **속성 ID**: IIS 메타베이스 설정의 숫자 속성을 지정합니다.  


### <a name="bkmk_regkey"></a> 레지스트리 키

- **Hive**: 검색 하려는 레지스트리 하이브를 선택 합니다.

    > [!Tip]  
    > **찾아보기** 를 선택 하 여 참조 컴퓨터의 값에서 설정을 구성 합니다. 원격 컴퓨터에서 레지스트리 키를 찾아보려면 원격 컴퓨터에서 **원격 레지스트리** 서비스를 사용 하도록 설정 합니다.  

- **키**: 검색할 레지스트리 키 이름을 지정합니다. `key\subkey` 형식을 사용합니다.  

- **이 레지스트리 키는 64비트 애플리케이션과 연결되어 있음**: 64비트 버전의 Windows를 실행하는 클라이언트에서 32비트 레지스트리 키 이외에 64비트 레지스트리 키를 검색합니다.  

    > [!NOTE]  
    > 동일한 64비트 컴퓨터의 64비트 및 32비트 레지스트리 위치에 같은 레지스트리 키가 모두 있는 경우 글로벌 조건을 통해 두 레지스트리 키가 모두 검색됩니다.  


### <a name="bkmk_regval"></a> 레지스트리 값

- **Hive**: 검색할 레지스트리 하이브를 선택 합니다.  

    > [!Tip]  
    > **찾아보기** 를 선택 하 여 참조 컴퓨터의 값에서 설정을 구성 합니다. 원격 컴퓨터에서 레지스트리 값을 찾아보려면 원격 컴퓨터에서 **원격 레지스트리** 서비스를 사용 하도록 설정 합니다. 또한 원격 컴퓨터에 액세스 하려면 관리자 권한이 필요 합니다.  

- **키**: 검색할 레지스트리 키 이름을 지정 합니다. `key\subkey` 형식을 사용합니다.  

- **값**: 지정된 레지스트리 키 내에 포함해야 하는 값을 지정합니다.  

- **이 레지스트리 키는 64비트 애플리케이션과 연결되어 있음**: 64비트 버전의 Windows를 실행하는 클라이언트에서 32비트 레지스트리 키 이외에 64비트 레지스트리 키를 검색합니다.  

    > [!NOTE]  
    > 동일한 64비트 컴퓨터의 64비트 및 32비트 레지스트리 위치에 같은 레지스트리 키가 모두 있는 경우 글로벌 조건을 통해 두 레지스트리 키가 모두 검색됩니다.  


### <a name="bkmk_script"></a> 스크립트

스크립트에서 반환되는 값은 글로벌 조건의 준수를 평가하는 데 사용됩니다. 예를 들어 VBScript를 사용하는 경우 **WScript.Echo Result** 명령을 사용하여 *Result* 변수 값을 글로벌 조건에 반환할 수 있습니다.  

- **검색 스크립트**: **스크립트 추가**를 선택 하 고 스크립트를 입력 하거나 찾습니다. 이 스크립트는 값을 찾는 데 사용 됩니다. Windows PowerShell, VBScript, 또는 Microsoft JScript 스크립트를 사용할 수 있습니다.  

- **재구성 스크립트 (선택 사항)** : **스크립트 추가**를 선택 하 고 스크립트를 입력 하거나 찾습니다. 이 스크립트는 비규격 설정 값을 재구성 하는 데 사용 됩니다. Windows PowerShell, VBScript, 또는 Microsoft JScript 스크립트를 사용할 수 있습니다.  

- **로그온된 사용자 자격 증명을 사용하여 스크립트 실행**: 이 옵션을 사용하도록 설정하면 로그인한 사용자의 자격 증명을 사용하는 클라이언트 컴퓨터에서 스크립트가 실행됩니다.  

> [!Note]  
> 버전 1810부터 Windows PowerShell을 검색 또는 재구성 스크립트로 사용할 경우 Configuration Manager 클라이언트는 `-NoProfile` 매개 변수를 사용 하 여 PowerShell을 호출 합니다. 이 옵션은 프로필 없이 PowerShell을 시작합니다. PowerShell 프로필은 PowerShell이 시작될 때 실행되는 스크립트입니다. <!--3607762-->  


### <a name="bkmk_sql"></a> SQL 쿼리

- **SQL Server 인스턴스**: SQL 쿼리를 기본 인스턴스, 모든 인스턴스, 지정된 데이터베이스 인스턴스 이름 중 어느 것에 대해 실행할지 선택합니다.  

    > [!NOTE]  
    > 인스턴스 이름은 SQL Server의 로컬 인스턴스를 참조해야 합니다. 클러스터형 SQL Server 인스턴스를 참조하려면 스크립트 설정을 사용해야 합니다.  

- **데이터베이스**: SQL 쿼리를 실행할 때 기준이 되는 Microsoft SQL Server 데이터베이스의 이름을 지정합니다.  

- **열**: 글로벌 조건의 호환성을 평가하는 데 사용할 Transact-SQL 문에서 반환되는 열 이름을 지정합니다.  

- **Transact-SQL 문**: 글로벌 조건에 사용할 전체 SQL 쿼리를 지정합니다. 기존 SQL 쿼리를 사용 하려면 **열기**를 선택 합니다.  

    > [!IMPORTANT]  
    > SQL 쿼리 설정에서는 데이터베이스를 수정하는 SQL 명령을 지원하지 않습니다. 데이터베이스에서 정보를 읽는 SQL 명령만 사용할 수 있습니다.  


### <a name="bkmk_wql"></a> WQL 쿼리

- **네임 스페이스**: 클라이언트 컴퓨터에서 호환성을 평가 하는 WMI 네임 스페이스를 지정 합니다. 기본값은 `root\cimv2`입니다.  

- **클래스**: 위 네임 스페이스의 대상 WMI 클래스를 지정 합니다.  

- **속성**: 위의 클래스에서 대상 WMI 속성을 지정 합니다.  

- **WQL 쿼리 WHERE 절**: 정규화 된 절을 지정 하 여 결과를 줄입니다. 예를 들어 Win32_Service 클래스에서 DHCP 서비스만 쿼리하려면 WHERE 절을 `Name = 'DHCP' and StartMode = 'Auto'`수 있습니다.   


### <a name="bkmk_xpath"></a> XPath 쿼리

- **경로**: 클라이언트 컴퓨터에서 호환성을 평가하는 데 사용할 .xml 파일의 경로를 지정합니다. Configuration Manager에서는 경로 이름에 모든 Windows 시스템 환경 변수 및 `%USERPROFILE%` 사용자 변수를 사용할 수 있습니다.  

- **Xml 파일 이름**: 위의 경로에서 xml 쿼리를 포함 하는 파일 이름을 지정 합니다.  

- **하위 폴더 포함**: 지정한 경로 아래의 하위 폴더도 모두 검색하려면 이 옵션을 사용하도록 설정합니다.  

- **이 파일은 64 비트 응용 프로그램에 연결 되어**있습니다. 64 비트 버전의 Windows를 실행 하는 Configuration Manager 클라이언트에서 `%Windir%\Syswow64` 32 비트 시스템 파일 위치 외 `%Windir%\System32` 64 비트 시스템 파일 위치를 검색 합니다.  

- **XPath 쿼리**: 올바른 전체 XPATH (XML 경로 언어) 쿼리를 지정 합니다.  

- **네임 스페이스**: XPath 쿼리 시 사용할 네임 스페이스와 접두사를 식별 합니다.  

암호화된 .xml 파일을 검색하려고 하면 준수 설정에서 해당 파일을 찾지만 XPath 쿼리에서 결과가 생성되지 않습니다. Configuration Manager 클라이언트에서 오류를 생성 하지 않습니다.  

XPath 쿼리가 올바르지 않으면 설정이 클라이언트 컴퓨터에서 비준수로 평가됩니다.  



##  <a name="configure-compliance-rules"></a>준수 규칙 구성  

규정 준수 규칙에는 구성 항목의 호환성을 정의 하는 조건을 지정 합니다. 설정의 준수를 평가하려면 먼저 하나 이상의 준수 규칙이 있어야 합니다. WMI, 레지스트리 및 설정 스크립트를 통해 정책을 준수 하지 않는 것으로 발견 하는 값을 재구성할 수 있습니다. 새 규칙을 만들 하거나 규칙을 선택 하려면 기존 설정을 모든 구성 항목에서 찾아볼 수 있습니다.  


### <a name="to-create-a-compliance-rule"></a>규정 준수 규칙을 만들려면  

1. **구성 항목 만들기 마법사**의 **준수 규칙** 페이지에서 **새로 만들기**를 선택합니다.  

2. 에 **규칙 만들기** 대화 상자에서 다음 정보를 제공 합니다.  

    - **이름**: 규정 준수 규칙의 이름을 입력합니다.  

    - **설명**: 규정 준수 규칙에 대한 설명을 입력합니다.  

    - **선택한 설정**: **찾아보기**를 선택하여 **설정 선택** 대화 상자를 엽니다. 규칙을 정의하려는 설정을 선택하거나 **새 설정**을 선택합니다. 작업을 완료했으면 **선택**을 선택합니다.  

        > [!Tip]  
        > 현재 선택한 설정에 대한 정보를 보려면 **속성**을 참조하세요.  

    - **규칙 유형**: 사용하려는 규정 준수 규칙의 유형을 선택합니다.  

        - **값**: 구성 항목에서 반환된 값을 지정하는 값과 비교하는 규칙을 만듭니다. 추가 설정에 대 한 자세한 내용은 [값 규칙](#bkmk_value)을 참조 하세요.  

        - **존재 여부**: 클라이언트 디바이스에 있는지 여부 또는 발견되는 횟수에 따라 설정을 평가하는 규칙을 만듭니다. 추가 설정에 대 한 자세한 내용은 [존재 rules](#bkmk_exist)를 참조 하세요.  

3. **확인**을 선택하여 **규칙 만들기** 대화 상자를 닫습니다.  




### <a name="bkmk_value"></a>값 규칙  

- **속성**: 확인할 개체의 속성은 선택한 설정에 따라 달라 집니다. 사용 가능한 속성은 설정 유형에 따라 달라 집니다. 

- **설정은 다음을 따라야**합니다.. 사용 가능한 규칙이 나 사용 권한은 설정 유형에 따라 달라 집니다.

- **지원되는 경우 비규정 준수 규칙 수정**: Configuration Manager가 비호환 규칙을 자동으로 수정하도록 하려면 이 옵션을 선택합니다. Configuration Manager는 다음 규칙 유형을 사용 하 여이 작업을 지원 합니다.  

    - **레지스트리 값**: 비규격 인 경우 클라이언트에서 레지스트리 값을 설정 합니다. 존재 하지 않는 경우 클라이언트에서 값을 만듭니다.  

    - **스크립트**: 클라이언트는 설정으로 지정한 재구성 스크립트를 사용 합니다.  

    - **WQL 쿼리**  

    > [!IMPORTANT]  
    > 규칙 연산자로 설정 된 경우에 비호환 규칙을 재구성할 수 있습니다 **Equals**.  

- **이 설정 인스턴스를 찾을 수 없는 경우 비 호환성 보고**:이 설정이 클라이언트 컴퓨터에 없는 경우 구성 항목에서 비준수를 보고 하도록 하려면이 옵션을 사용 하도록 설정 합니다.  

- **보고할 비호환성 심각도**: 이 호환성 규칙이 실패하는 경우 Configuration Manager 보고서에 보고되는 심각도 수준을 지정합니다. 사용할 수 있는 심각도 수준은 다음과 같습니다.  
    - **없음**  
    - **정보**  
    - **경고**  
    - **위험**  
    - **위험(이벤트 포함)** : 이 호환성 규칙에 실패한 컴퓨터는 **위험**의 오류 심각도를 보고합니다. 또한 이 심각도 수준은 애플리케이션 이벤트 로그에 Windows 이벤트로 기록됩니다.
